# PixelPrism Documentation

This directory contains the MkDocs project that ships the PixelPrism documentation.
The site uses [Material for MkDocs](https://squidfunk.github.io/mkdocs-material/)
with a [Catppuccin](https://github.com/catppuccin) color palette override.

## Building the Documentation

To build the documentation, install the documentation dependencies first:

```bash
# Install the package with documentation dependencies
pip install -r docs/requirements.txt
pip install -e .
```

Then you can build the documentation using MkDocs:

```bash
# Build the documentation
mkdocs build

# Or serve it locally
mkdocs serve
```

## Documentation Structure

- `index.md`: Main landing page and links to the rest of the content.
- `guides/`: High-level guides (`overview`, `expressions`, `animation`).
- `examples.md`: Cookbook-like scenarios.
- `types.md`: Dataclasses, enums, and type aliases.
- `reference/`: Auto-generated API reference (populated during `mkdocs build`).
- `gen_ref_pages.py`: Builds one Markdown file per module using `mkdocs-gen-files`.
- `stylesheets/catppuccin.css`: Overrides Material tokens with Catppuccin Latte + Mocha colors.

## How Documentation is Generated

The documentation is generated using:

1. **mkdocs** – Static site builder.
2. **mkdocs-material** – Provides the base theme, navigation behavior, and search UI.
3. **Catppuccin palette override** – Custom CSS layer to align with the Catppuccin design.
4. **mkdocstrings** – Pulls docstrings directly from the PixelPrism package.
5. **mkdocs-gen-files** – Writes module reference stubs at build time.
6. **mkdocs-literate-nav** – Reads the generated `reference/SUMMARY.md` and turns it into a nav tree.
7. **mkdocs-autorefs** – Ensures symbol cross-references are resolved automatically.

The `gen_ref_pages.py` script scans the `pixelprism` package and generates Markdown
files for each module right before MkDocs renders the site. The resulting files are
not committed; they only live in the in-memory build and are served/generated by
`mkdocs serve` or `mkdocs build`.

## Adding New Documentation

To add new documentation:

1. Add or update docstrings in the code (they drive the API reference).
2. Create new Markdown files in `docs/`.
3. Update the `nav` section in `mkdocs.yml` if needed.
4. Run `mkdocs serve` to preview changes locally.
